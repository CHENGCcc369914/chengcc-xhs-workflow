# 创作路由

## 默认路由

用户只给主题时，按下面步骤执行：

1. 主题归类：情绪 / 职场 / 关系 / AI / 经典方法论 / 个人品牌。
2. 写作前先做本地低粉高爆案例库 RAG 检索。runtime/template 运行时使用 active creator profile 的 `paths.ragCaseLibrary`；Cc 私有安装且没有 profile 时，才使用 Cc 本地案例库 fallback。
3. 从相似案例中提取：标题钩子、开头真实铺垫、脑内台词、AI 转折问题、方法清单、温柔总结、收藏 CTA。
4. 压缩成一个小道理。
5. 先判断有没有朋友吐槽式破案入口：具体小现场、反差、读者自我映射和评论话口。
6. 选择 1-3 位对标博主。
7. 选择正文公式：朋友吐槽式破案 / 心理动作拆解 / 职场反思 / AI 工具不讲工具。
8. 构造一个真实或模拟场景。
9. 生成四列 AI 复盘表。
10. 生成完整小红书内容。
11. 如果是图文卡片，读取 `05-visual-and-carousel-style.md`，压成 4 页结构并拆清图片/正文分工。
12. 做发布前风险检查。

如果用户说“按中控台确认版”“完整工作流”“先给选题”“做成 Skill”“4 页图文”“Image 2”“发布审核”“手动发布清单”，不要直接写最终稿。改由 `chengcc-xhs-workflow` 主导完整工作流；本 Skill 只提供澄Cc账号声音、标题、正文、AI 复盘动作和内容风险判断：

1. 读取默认资料和中控台确认版。
2. 先输出 10 个适合账号的选题。
3. 等用户从 10 个里选 1 个。
4. 再做 4 页图文、正文、标签和 Image 2 完整卡片 prompt。
5. 等用户确认内容。
6. 再生成图片或输出出图 prompt；如已生成图片，必须创建本次 `run-manifest.json`，记录 run_id、卡片文案 hash、源图路径、导出路径、视觉/IP 验收和 rejected attempts。
7. 只有 manifest 里每页 `status: pass` 且 `publish_ready: true` 的导出图可以进入上传顺序；旧图、错风格图、半写实拼贴、旧橙子 mascot、文字不匹配图必须排除。
8. 按 2026 发布审核门检查 AI 标识、真实性、原创版权、隐私、承诺和平台风险。
9. 审核通过后输出手动发布清单。

## 主题到结构

| 主题类型 | 适合标题 | 适合结构 | 结尾动作 |
|---|---|---|---|
| 情绪内耗 | 建议大家不要... | 心理动作拆解 | 事实/解释分离表 |
| 职场反思 | 新手最该练的不是 X，而是 Y | 职场反思 | 模拟汇报/表达 prompt |
| 关系边界 | 不是 A，而是 B | 心理动作拆解 | 课题分离 prompt |
| AI 复盘 | 用 AI 复盘后才发现... | AI 工具不讲工具 | 可复制 prompt |
| 经典方法论 | 古人这句话我以前理解错了 | 场景 + 误解 + 重读 | 一个今天能做的小动作 |
| 个人品牌 | 先完成再完美不是摆烂 | 新手进阶记录 | 低门槛输出模板 |

默认新增一层判断：

| 判断项 | 通过标准 | 不通过时 |
|---|---|---|
| 可围观现场 | 像朋友会想听的真实小事 | 先补具体场景 |
| 反差 | 有“我以为 A，结果 B” | 重写标题和开头 |
| 自我映射 | 读者能说“我也这样” | 加脑内台词或生活细节 |
| 评论话口 | 能回答选择题/经历题 | 不进入最终稿 |

## 信息不足时

不要先停下来问很多问题。先用默认定位写一个可用版本，然后在末尾给：

- 如果加入真实经历，可以补充哪 3 个点。
- 如果要更像某位博主，可以调整哪 2 个方向。
- 下一篇可以复用到哪个场景。

只在以下情况提问：

- 用户要求使用真实经历，但没有给任何场景。
- 用户要求发布到小红书后台。
- 内容涉及医疗、心理治疗、法律、投资、职业承诺等高风险判断。

## 小红书四件套路由

当多个小红书 Skill 都可能触发时，先按任务类型分工：

1. `xhs-blogger-intelligence`：对标博主监测、watchlist、公开笔记采集、note-card、RAG brief。
2. Active creator profile 的 RAG 案例库：低粉高爆案例、类似选题表达、爆文正文结构、写作前 RAG 检索的第一资料源。
3. `chengcc-content-loop-runtime`：默认澄Cc单篇生产和跨篇闭环。只要用户说“做一篇澄Cc小红书 / 用澄Cc默认资料做一篇 / 帮我写一篇澄Cc小红书 / 做一个澄Cc选题”，先建 run record。
4. `chengcc-xhs-workflow`：runtime 内的单篇发布包、10 个选题、4 页图文、Image 2、发布审核、手动发布清单。
5. `cc-xhs-personal-growth-writer`：澄Cc账号声音、标题、正文、封面文案、AI 复盘动作；只有用户明确说“只写文案 / 不建 run / 不跑流程 / 轻量草稿”时才独立主导。
6. `xiaohongshu-ops`：真实小红书浏览、搜索、上传、填发布页、评论回复和后台操作。

优先级：

- 有真实平台动作时，`xiaohongshu-ops` 负责平台操作。
- 普通澄Cc小红书生产默认先走 `chengcc-content-loop-runtime`，即使用户只说“写一篇”。
- 有对标博主最新动态或 RAG 需求时，先跑 `xhs-blogger-intelligence`。
- 有低粉高爆案例参考需求时，优先读取 active creator profile 的 RAG 案例库，再生成 RAG brief。
- 有完整发布包、图片生成、审核、手动发布清单或复盘时，`chengcc-content-loop-runtime` 是父入口，`chengcc-xhs-workflow` 是单篇生产子流程。
- 只有用户明确要求写作、改文案、标题、正文，且明确不建 run / 不跑流程时，本 Skill 是主入口。

## 发布协作路由

当用户要“做图文卡片并发布”时：

1. `chengcc-content-loop-runtime` 先创建或继续 run，读取最近经验，并保留 Gate 1 / Gate 2。
2. `chengcc-xhs-workflow` 负责 run 内的完整发布包、Image 2 prompt / 生成状态、run manifest、发布审核和手动发布清单。
3. 本 Skill 负责给完整流程提供澄Cc账号声音、标题、正文、AI 复盘动作和内容风险判断。
4. Image 2 负责生成最终完整卡片，中文文字直接在图里完成。
5. 发布前必须有 manifest 验收门和平台审核门；有问题先改，不进入发布。
6. 手动发布清单只能列 `run-manifest.json` 标记 `pass` 的导出图，不能列 rejected、reference_only、fail 或旧目录图片。
7. Cc 手动进入小红书创作者平台上传图片、填写标题正文标签，并自行决定是否点击发布。
8. Codex 不再默认硬控 Creator Center 填文案；只有用户明确要求时才尝试浏览器操作，并按 `xiaohongshu-ops` 的平台规则执行。

## 一题多写

当用户要求“拆成多个选题”时，输出每条：

- 标题。
- 场景。
- 核心观点。
- AI 使用动作。
- 评论区问题。
- 参考博主结构。

默认拆 8 条，覆盖：

1. 职场汇报。
2. 被领导评价。
3. 朋友关系。
4. 亲密关系。
5. 自我怀疑。
6. 拖延。
7. AI 复盘。
8. 个人品牌写作。
